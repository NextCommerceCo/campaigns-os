# Adopt Normalized Design Source Packages

Campaign design inputs can arrive as template-stock source, Figma-driven exports, AI-generated HTML, hand-authored HTML, existing Page Kit pages, agency-produced static source, or a mixture of those sources. We will keep those inputs flexible, but every normal Campaigns OS build will produce a public Design Source Package that normalizes provenance, explicit Contribution Coverage, a campaign-level Surface Identity catalog, contribution-level Page Kit mapping, comparison references, Source Gaps, Source TODOs, and source divergences before Build and Polish consume it.

This keeps agency workflows open while giving the lifecycle machine-readable creative context. The Design Source Package remains separate from CampaignSpec commerce truth, the Campaign Build Brief's merchandising interpretation, Template Reference baselines, and the Polish Evidence Package; those artifacts are linked by fingerprints and stable Surface Identity rather than collapsed into one loose handoff.

The Design Source Package is a new public artifact alongside existing producer contracts. Existing source-html manifests remain producer/source-HTML adapter inputs; `prepare-build` can use them to emit Design Source Contributions, but we do not rename or stretch the manifest into the broader package. During v0 compatibility, `prepare-build` may synthesize a package from legacy packet/source-html data when no package exists, but downstream Build and Polish still consume the package concept rather than maintaining a second legacy source model. The package lives at `.campaign-runtime/input/design-source-package.json` by default and is referenced from the Build Packet, Build Context, and Assembly Report by path, full artifact hash, and material fingerprint instead of embedded wholesale. The full hash supports audit and reproduction; the material fingerprint drives freshness gates. Its schema is `campaign-design-source-package/v0` in `schemas/campaign-design-source-package.v0.schema.json`; it stays outside the `campaign-runtime-*` namespace because it is upstream creative/provenance context, not runtime assembly proof.

The Design Source Package owns source/reference screenshot refs only: canonical URLs, exports, captured source renders, or explicit unavailable-render notes. Polish Evidence owns built-output and comparison screenshots for a specific build fingerprint, and compares against source refs and Template Reference refs without rewriting those source artifacts. If Polish can capture a missing canonical source render, it proposes a source-reference update or Source TODO; source preparation or an explicit source-reference refresh action owns package mutation and records attribution. Material source-reference refreshes create a new Design Source Package fingerprint and make downstream evidence tied to the old source fingerprint stale; material fields include contribution identity, provenance, presentation intent, Surface Identity, coverage mappings, source refs, screenshot/reference refs, Template Reference linkage, Source Gaps, Source TODOs, accepted waivers, and readiness-affecting divergences or exceptions. Administrative edits may be non-material only when they do not alter readiness, coverage, provenance, or comparison basis. Renderable page-level `primary_design` contributions require at least desktop and mobile source screenshot refs for source readiness; tablet is optional in v0. Missing required captures become Source TODOs unless accepted as Source Gaps or covered by approved waivers. Template-baseline pages use Template Reference standard viewport refs. Source refs, Polish Evidence, and QA evidence use shared viewport keys (`mobile`, `desktop`, optional `tablet`), while exact dimensions and device/capture details remain artifact metadata.

Surface Identity starts with page-level mappings during source preparation. Build may refine child section or runtime-surface identities as Page Kit output takes shape, while Polish and QA attach evidence or observations to the established vocabulary instead of minting new pages. `campaign` is a reserved Surface Identity with `kind: "campaign"` for campaign-level gaps, TODOs, waivers, and readback references. Surface Identity IDs are stable human-semantic strings with labels and aliases; opaque source or DOM IDs remain external references. Page-level Surface Identity is distinct from CampaignSpec/Map Builder page IDs, custom page names, public routes, producer page types, and Campaign Page Kit runtime `page_type`, which remain mapped attributes or aliases. Stable human-readable CampaignSpec page IDs may seed primary page-level IDs; otherwise source preparation derives IDs from normalized page role plus order. `surface_identity[]` is a structured catalog (`id`, `kind`, `label`, `aliases`, `mappings`), not a bare string list. Contribution mappings reference canonical Surface Identity IDs and carry relationship metadata; they do not define competing page/spec/Page Kit mappings. `coverage_role` is a small enum (`primary_design`, `partial_design`, `brand_tokens`, `asset_source`, `copy_source`, `template_baseline`, `reference_only`, `fallback_legacy`) with notes for unusual cases. Mapping `confidence` is `high`, `medium`, `low`, or `unknown`, describing confidence in the source-to-surface relationship rather than quality or approval. Low confidence blocks source readiness only when it affects required page-level primary design coverage; otherwise it is carried as a gap, note, or readback signal. Required page-level coverage applies to active or mapped pages in the current build scope: each page needs non-low-confidence `primary_design`, an explicit Template Reference-backed `template_baseline` for template-stock pages, or an attributed Source Gap / approved Checkpoint Waiver explaining the missing primary source.

The package also supports resumable Readiness Checkpoints. Campaigns OS may still complete in one session when inputs are prepared cleanly, but the normal workflow can pause and resume as gaps, TODOs, waivers, build evidence, polish evidence, deploy state, and QA proof become explicit.

The Design Source Package checkpoint can complete as `ready_with_gaps` when Source Gaps are explicit and attributed. Source TODOs remain blocking preparation work unless waived for the run.

The package carries a generated top-level `readiness` summary for cheap orchestration and readback (`status`, `blocking_reasons`, `gap_count`, `todo_count`, `waiver_count`, `generated_at`), while detailed gaps, TODOs, and waivers remain authoritative. Source readiness does not use `ready_with_warnings`; warning-like conditions must be represented as concrete gaps, TODOs, proposed exceptions, notes, or waivers. Free-form notes are allowed for context, but they do not affect readiness. Source gaps and TODOs carry `scope` and `applies_to`; they should attach to Surface Identity when possible, including the reserved `campaign` identity for legitimate campaign-level concerns.

Readiness Checkpoints use a shared status vocabulary (`pending`, `blocked`, `ready`, `ready_with_gaps`, `ready_with_waivers`, `completed`, `completed_with_warnings`, `skipped`) so stage-specific complexity lives in evidence and next actions instead of status-name drift.

`ready_with_waivers` is only a summary status. Each waiver remains a structured, attributed, time-bound artifact with scope, applies-to references, reason, owner, and expiry or review condition.

Polish may recommend or draft a waiver with screenshots and findings, but it does not approve the waiver itself; approval is an operator/run decision because it changes readiness.

Polish Evidence must classify unresolved issues as `repair_needed`, `source_gap`, `source_divergence`, `waiver_recommended`, or `out_of_scope` so resumed runs do not inherit an ambiguous issue bucket. `repair_needed` blocks deploy and QA unless the issue is repaired, reclassified, or covered by an approved waiver. Polish Evidence is current only when it matches both the current build fingerprint and the current Design Source Package material fingerprint; if either changes materially, the evidence is stale unless a structured waiver explains the exception. Assembly also records the Design Source Package material fingerprint it consumed; if that Assembly value is missing or stale against the current package, `campaigns-os next` routes back to Build before Polish. A stale Assembly source fingerprint can be waived only by an explicit, scoped, attributed Source Freshness Waiver; that waiver allows Polish to proceed but does not waive missing Polish Evidence, and it must stay visible in readiness readback and QA evidence. During v0, accepted Source Freshness Waivers are written directly into `waivers[]`; a future generic waiver CLI should cover all Checkpoint Waivers rather than becoming a source-specific command. During the v0 transition, the gate enforces the source-package fingerprint only when the Assembly Report exposes a current Design Source Package material fingerprint; legacy reports without one keep the build-fingerprint gate and emit a readiness warning.

A Source Divergence discovered by Polish starts as proposed. It becomes accepted only through an operator/run decision or confirmation by the relevant Build or Design Source owner.

A Source Gap discovered by Polish also starts as proposed unless it traces to an accepted gap already present in the Design Source Package.

The Design Source Package carries a generated human-readable readback summary for session pickup, but structured fields remain authoritative. Source preparation owns that readback; later stages write their own stage readbacks or evidence summaries rather than rewriting source provenance. Stage readbacks are surfaced through a consolidated or just-in-time Campaign Readiness Readback so operators are not forced to discover a patchwork of artifacts. The readiness readback is generated live from current artifacts as the primary behavior, with optional Run Record snapshots for audit. `campaigns-os next` shows the concise current-stage readback; run or campaign status shows the fuller campaign-level readback. The readback uses stable structured buckets plus prose so humans, agents, and future UI can present the same facts consistently. It references evidence rather than embedding proof details, and screenshots remain refs with optional UI thumbnails.
